# Glossary

| Term | Meaning |
|---|---|
| Requirement | A documented need or constraint that a solution must satisfy. |
| Business requirement | A business outcome, policy, or capability need. |
| Functional requirement | A behavior the system must perform. |
| Non-functional requirement | A quality attribute such as performance, security, availability, auditability, or usability. |
| User story | A backlog item describing value from a user, business, or system perspective. |
| Acceptance criteria | Testable conditions that define when a story or requirement is acceptable. |
| Backlog | Ordered set of work items for a product or delivery team. |
| Epic | Large body of work that can be split into features or stories. |
| Feature | Coherent capability that supports an epic or product objective. |
| Dependency | A required system, team, decision, data set, environment, approval, or event. |
| Assumption | A working belief that has not yet been confirmed. |
| Risk | An uncertain condition that could affect outcome, quality, cost, time, or operations. |
| Issue | A known problem requiring action. |
| Decision | A choice made by an accountable person or group. |
| Traceability | Linkage between business need, requirement, story, acceptance criteria, test, defect, and decision. |
| UAT | User Acceptance Testing performed by business or operational representatives. |
| Release readiness | Evidence-based assessment that a change is ready to go live. |
| Impact analysis | Assessment of what a change affects across systems, data, process, users, operations, and delivery. |
| Change request | Proposed modification to agreed scope, requirement, schedule, or implementation approach. |
| Definition of Ready | Criteria that indicate a backlog item is ready for delivery planning or sprint selection. |
| Definition of Done | Criteria that indicate work is complete and accepted by the team. |
